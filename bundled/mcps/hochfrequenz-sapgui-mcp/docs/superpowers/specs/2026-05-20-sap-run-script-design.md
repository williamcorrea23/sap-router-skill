# Design: `sap_run_script` — sandboxed Python scripting over SAP GUI COM

**Date:** 2026-05-20
**Status:** Approved, pending implementation

---

## Problem

`sap_com_evaluate` lets an LLM batch flat COM operations (get/set/call) in a single round-trip, but
has no control flow. Multi-step SAP workflows that require loops ("process every row in this ALV
grid") or conditional branching ("if status == 'Error' then…") still require many sequential tool
calls — one per decision point — which is slow and error-prone.

Two root causes:
1. **Round-trip cost** — each LLM tool call is a separate MCP request/response cycle.
2. **Logic reliability** — the LLM must produce a correct sequence of tool calls up-front, with no
   ability to branch based on runtime SAP responses.

---

## Solution

A new MCP tool `sap_run_script` that accepts a Python script string, executes it inside a
restricted namespace on the COM thread, and returns collected output + any error.

The script gets a live **sapsucker `GuiSession`** object and an `output()` function. It can use
full Python control flow (loops, conditionals, try/except, list comprehensions). Results are
returned in one MCP response when the script finishes.

---

## Threat model (explicit)

### What we defend against

**Accidental damage from LLM-generated code.** The primary risk is the LLM writing something like
`import os; os.remove(...)` or `import subprocess; subprocess.run(...)` out of habit or
hallucination. This is addressed by replacing `__builtins__` entirely — `__import__` is not in
scope, so any `import` statement raises `NameError` before it can reach the OS.

### What is explicitly out of scope

**Deliberate sandbox escape via Python class-hierarchy traversal.** A motivated attacker can reach
`subprocess.Popen` through `"".__class__.__mro__[-1].__subclasses__()` even without `import`. We
do **not** block this. The reasoning:

- The scripts are generated by an LLM operating under a system prompt focused on SAP automation.
  Such an LLM will never spontaneously write MRO traversal to reach subprocess.
- This is an internal developer tool, not a publicly exposed endpoint.
- Blocking MRO traversal correctly requires RestrictedPython (or equivalent AST rewriting), which
  adds a dependency and maintenance burden disproportionate to the actual risk.
- If the threat model hardens (e.g., scripts submitted by untrusted external users), the migration
  path to RestrictedPython is straightforward: swap out the `exec()` call and keep everything else.

## Design

### Tool signature

```python
async def sap_run_script(
    script: str,           # Python source — the script body (not a function definition)
    session: str | None,   # Session ID, e.g. "s1". None → primary session.
    agent_id: str | None,  # Agent identifier for binding check. Optional.
) -> SapRunScriptResult
```

Desktop backend only (same gate as `sap_com_evaluate`).

`ToolAnnotations`: `destructiveHint=True`, `openWorldHint=False` — scripts can write SAP data, and
the scope is limited to the local desktop COM session.

### Script namespace

```python
SAFE_BUILTINS = {
    # Iteration / sequences
    "range": range, "len": len, "enumerate": enumerate, "zip": zip,
    "map": map, "filter": filter,
    "sorted": sorted, "reversed": reversed,
    "list": list, "tuple": tuple, "dict": dict, "set": set,
    # Numeric / string
    "int": int, "float": float, "bool": bool, "str": str,
    "min": min, "max": max, "abs": abs, "round": round, "sum": sum,
    # Logic
    "any": any, "all": all, "isinstance": isinstance,
    # Exception types (needed for scripts to raise typed exceptions)
    "Exception": Exception, "ValueError": ValueError, "KeyError": KeyError,
    "TypeError": TypeError, "IndexError": IndexError, "AttributeError": AttributeError,
    "RuntimeError": RuntimeError, "StopIteration": StopIteration,
    "NotImplementedError": NotImplementedError,
    # True / False / None are Python keywords in Python 3; listing them here is harmless
    # but not functionally necessary — they resolve without going through __builtins__.
}

exec(compile(script, "<sap_script>", "exec"), {
    "__builtins__": SAFE_BUILTINS,
    "session": desktop_session,   # sapsucker GuiSession — full SAP GUI typed API
    "output": output_fn,          # output(x) → appends x to result list
})
```

**Not provided (accidental-use defence):** `open`, `eval`, `exec`, `print`, `input`, `compile`,
`globals`, `locals`, `breakpoint`, `setattr`, `delattr`, `hasattr`, `vars`, `dir`,
and all stdlib modules.

Note: `Exception` and its common subclasses ARE provided — scripts must be able to raise typed
exceptions.

**`__import__` requires a special stub.** When `__builtins__` is a plain dict (not the builtins
module), CPython's import opcode raises `ImportError: __import__ not found` if the key is absent —
not `NameError`. To satisfy the spec's security contract (`import os` → `NameError`), include a
`_blocked_import` stub that raises `NameError` explicitly:

```python
def _blocked_import(*args, **kwargs):
    raise NameError("__import__ is not available in sap_run_script scripts")

SAFE_BUILTINS["__import__"] = _blocked_import
```

> **Note:** `getattr` is provided so LLM-generated scripts can use dynamic attribute access
> (`getattr(obj, attr_name)`), analogous to `isinstance`. MRO traversal
> (`obj.__class__.__mro__`) is not blocked — see Threat model.

### Output coercion

`output_fn` coerces each value to a JSON-safe type before appending, so that COM dispatch proxies
(and other non-serialisable objects) never reach FastMCP's response serialiser:

```python
def output_fn(value: Any) -> None:
    try:
        json.dumps(value)
        collected.append(value)
    except (TypeError, ValueError):
        collected.append(str(value))
```

`collected` is a local list closed over by `output_fn` inside `_run_script()`. It is created and
consumed entirely on the COM thread — no cross-thread mutation.

### Execution model

The script runs inside `com.run(lambda)` on the **dedicated COM thread**. This is required because
SAP GUI COM uses Single-Threaded Apartment (STA) — all calls to a session must originate from the
thread that initialised the COM apartment.

`backend.require_session()` reads a `ContextVar` and **must be called on the async side** before
the `com.run()` lambda — same pattern as `sap_com_evaluate`. Calling it inside the lambda reads
the wrong context and must not be done.

**Compile-time errors** (`SyntaxError` from `compile()`) are caught in the same `except Exception`
block as runtime errors and returned as `success=False` — they never propagate past the tool
boundary. Note: Python 3 raises `SyntaxError` (not `ValueError`) for null bytes in source strings.

**Partial results are preserved:** if the script calls `output()` 80 times and then raises on the
81st iteration, the 80 collected items are returned alongside the error.

### No hard timeout (v1 limitation)

Windows STA prevents cross-thread interruption of COM calls. An infinite loop in the script will
block the COM thread until the MCP transport times out. This is accepted for v1. A mitigation
(`sys.settrace`-based instruction counter) can be added later if it proves necessary in practice.

### Return model — `SapRunScriptResult`

`SapRunScriptResult` inherits **`ToolResult`** from `sapguimcp.models.base`. The inherited
`popup` and `active_window` fields are never populated and serialise to nothing (FastMCP omits
`None` fields). Inheriting `ToolResult` reuses its model validator and `failure()` classmethod
without duplication.

The inherited `failure()` signature is:
```python
@classmethod
def failure(cls, error: str, **kwargs: Any) -> Self: ...
```
Use `**kwargs` to pass partial results: `SapRunScriptResult.failure("NameError: ...", output=collected)`.

| field | type | notes |
|---|---|---|
| `success` | `bool` | `True` if script completed without raising |
| `output` | `list[Any]` | Everything passed to `output()`, in call order (JSON-safe by construction) |
| `error` | `str \| None` | `ExceptionType: message` if script raised or `compile()` failed |
| `error_traceback` | `str \| None` | Populated via `traceback.format_exc()` in the except block; default `None` on success |

When `success=True` and `output=[]` (script completed but never called `output()`), the result is
returned as-is. A DEBUG-level log line is emitted: `"sap_run_script: script completed with no
output"`. The tool description instructs the LLM to always call `output()` at least once with a
summary.

**Tool description (draft for implementer):**

> Execute a Python script against the live SAP GUI session (desktop backend only).
>
> The script receives a `session` variable (sapsucker `GuiSession`) and an `output(value)` function.
> Call `output()` to collect results — all values passed to it are returned in order.
> Always call `output()` at least once with a summary; a script that never calls `output()`
> returns an empty list with no indication of what happened.
>
> `import` and `print` are not available. Use `output()` to collect results, not `print()`.
> Use `session.find_by_id(id)` to reach elements, then read/write their properties and call
> their methods directly. Full Python control flow (loops, if/else, try/except) works normally.
>
> **When to use this tool vs `sap_com_evaluate`:** prefer `sap_com_evaluate` for fixed-step
> sequences (known number of operations). Use `sap_run_script` when the number of operations
> depends on a runtime value — e.g. iterating all rows in a grid, scanning tree nodes, or
> branching based on a field value read mid-sequence.
>
> If the script raises an unhandled exception, `success=False` and `output` contains whatever was
> collected before the error — partial results are preserved.
> **On failure:** inspect `error` (exception type + message) and `error_traceback` to understand
> what went wrong, then either fix the script and retry, or fall back to `sap_com_evaluate` for
> individual operations. If the error mentions an element ID that was not found, call
> `sap_com_snapshot` first to verify the correct ID.
>
> Example:
> ```python
> grid = session.find_by_id("wnd[0]/usr/cntlGRID/shellcont/shell")
> errors = [grid.get_cell_value(r, "VBELN") for r in range(grid.row_count)
>           if grid.get_cell_value(r, "STATUS") == "Error"]
> output({"error_count": len(errors), "docs": errors})
> ```

## Example

```python
# Script sent by LLM — processes every error row in an ALV grid
grid = session.find_by_id("wnd[0]/usr/cntlGRID/shellcont/shell")
errors = []
for row in range(grid.row_count):
    status = grid.get_cell_value(row, "STATUS")
    if status == "Error":
        doc = grid.get_cell_value(row, "VBELN")
        errors.append(doc)
output({"error_count": len(errors), "docs": errors})
```

Single MCP round-trip replaces N separate `sap_com_evaluate` calls.

## Acceptance criteria

- [ ] Script that calls `output(x)` returns `success=True`, `output=[x]`, `error=None`
- [ ] Script that raises returns `success=False`, `error` contains exception type + message,
      `output` contains whatever was collected before the raise
- [ ] Script with `import os` raises `NameError` (not `ImportError`) — central security contract
- [ ] Script with a syntax error returns `success=False`, `error` starts with `"SyntaxError"`;
      a script with null bytes returns `success=False`, `error` starts with `"SyntaxError"`
      (Python 3 raises `SyntaxError`, not `ValueError`, for null bytes)
- [ ] Script that completes without calling `output()` returns `success=True`, `output=[]`
- [ ] `output()` called with a non-serialisable value (e.g. a MagicMock COM object) coerces to
      `str()` and does not raise
- [ ] When `backend.backend_type != "desktop"`, the tool returns `SapRunScriptResult.failure(...)`
      with a descriptive message — same gate pattern as `sap_com_evaluate`
- [ ] `session=None` resolves to the primary session (same as all other desktop tools)

---

## Out of scope (v1)

- Hard execution timeout
- `await` in scripts (synchronous only — runs on COM thread)
- Script caching or persistence between calls
- WebGUI backend support
- RestrictedPython upgrade (deferred unless threat model hardens)
