# GR/IR 정리(GR/IR Clearing) IMG 구성 가이드

## SPRO 경로
```
SPRO → Financial Accounting → Accounts Payable
    → Invoice Verification → GR/IR Clearing
```

또는

```
SPRO → Materials Management → Purchasing
    → Purchase Order → Invoice Verification → GR/IR Clearing Account
```

## 필수 선행 구성

- [ ] OB13: GL 계정 차트 정의 (중간 계정 생성)
- [ ] OX02: 회사코드 정의
- [ ] OBYA: 계정결정 기본 설정
- [ ] OMR6: GR/IR 허용범위 설정

## GR/IR 개념

**GR/IR (Goods Receipt/Invoice Receipt)**:
- GR(Goods Receipt): 상품 입고 시점
- IR(Invoice Receipt): 송장 수령 시점
- **차이**: GR와 IR 날짜가 다를 때 발생하는 임시 계정

### 시나리오: GR/IR 발생 과정

```
Timeline 1:
  2024.01.15: 상품 입고 (GR)
    Debit  530000 (자재비)
    Credit 199000 (GR/IR 중간계정) — 선 기인

Timeline 2:
  2024.02.05: 송장 수령 (IR) — 21일 차이
    Debit  199000 (GR/IR 중간계정) — 선 제거
    Credit 200000 (채무)

결과: GR/IR 계정은 임시 (최종 0)
```

## GR/IR 구성의 핵심 T-codes

### 1. OBYP — GR/IR 자동 정리 설정(GR/IR Account Assignment)

**SPRO 경로**: `SPRO → Financial Accounting → Accounts Payable → Invoice Verification → Automatic GR/IR Clearing`

**목적**: 송장 입력 시 GR/IR 계정을 자동으로 상쇄하는 규칙 정의

**구성 단계** (T-code: OBYP):

1. **GR/IR 계정 정의**
   ```
   회사코드: 1000
   국가: "KR"
   GR/IR 계정: 199000 (자산 — 유동)
   오프셋 계정: 200000 (채무)
   ```

2. **자동 정리 규칙**
   ```
   거래 조건: 
   - GR/IR 계정 잔액이 있으면
   - 송장 입력 시 자동으로 상쇄
   - 정확히 일치하는 금액만 정리 (기본)
   ```

3. **허용 범위 설정** (OMR6)
   ```
   상한: 100,000 원 (또는 금액의 1%)
   - 100,000원 이하 차이는 자동 정리
   - 초과 차이는 수동 처리
   ```

**필드 설명**:

| 필드 | 설명 | 값 | 용도 |
|------|------|-----|------|
| Company Code | 회사코드 | 1000 | 필수 |
| GR/IR Account | GR/IR 중간계정 | 199000 | 임시 계정 |
| Offsetting Account | 오프셋 계정 | 200000 | 최종 상쇄 |
| Clearing Method | 정리 방식 | Auto (자동) | 수동 허용시 Manual |
| Tolerance | 허용 범위 | 100000 | OMR6와 연계 |

---

### 2. MR11 — GR/IR 정리 수동 실행(Clear GR/IR Account)

**SPRO 경로**: `SPRO → Materials Management → Purchasing → Purchase Order → Invoice Verification → Clear GR/IR Account`

**목적**: 자동으로 정리되지 않은 GR/IR 항목을 월말에 수동으로 정리

**수동 정리 절차** (T-code: MR11):

1. **정리 대상 선택**
   ```
   회사코드: 1000
   기간: 202401 (2024년 1월)
   정리 기준: 
   - 날짜 선택: 30일 이상 차이
   - 금액 선택: 100,000원 초과
   ```

2. **정리 방식 선택**
   ```
   방식 A: 자동 정리 (GR와 IR 일치하는 것만)
   방식 B: 수동 정리 (사용자 확인 후)
   방식 C: 전기 없음 (검토만)
   ```

3. **실행 및 결과**
   ```
   실행 버튼 → GR/IR 항목 조회
   
   결과 (자동 정리 성공 시):
   - GR/IR 잔액: 0 원
   - 생성된 전기: 
     Debit  200000 (채무) X,XXX원
     Credit 199000 (GR/IR) X,XXX원
   ```

**구성 검증** (T-code: FB03 또는 FS10):
```
T-code: FS10 (GL Account Balance)
계정: 199000 (GR/IR)
기간: 202401
결과: 정리 후 0 또는 소액 (허용 범위 내)
```

---

### 3. OMR6 — GR/IR 허용한도 설정(GR/IR Tolerance)

**SPRO 경로**: `SPRO → Materials Management → Purchasing → Purchase Order → Invoice Verification → Set GR/IR Tolerance`

**목적**: 자동 정리 시 허용할 금액/비율 상한선 정의

**구성 단계** (T-code: OMR6):

1. **허용범위 설정**
   ```
   평가 기준:
   - 절대금액: 100,000 원 (모든 거래)
   - 비율: 1% (PO 금액 대비)
   
   예시:
   PO 금액: 1,000,000 원
   허용범위: 1% = 10,000 원
   
   GR/IR 차이 5,000 원 → 자동 정리 가능
   GR/IR 차이 15,000 원 → 자동 정리 불가 (초과)
   ```

2. **공급업체별 차등 설정** (선택)
   ```
   공급업체 Tier 1 (신뢰도 높음):
   - 허용한도: 2% 또는 200,000 원
   
   공급업체 Tier 2 (일반):
   - 허용한도: 1% 또는 100,000 원
   
   공급업체 Tier 3 (신뢰도 낮음):
   - 허용한도: 0.5% 또는 50,000 원
   ```

3. **가변비 설정** (K0)
   ```
   GR/IR 정리 구분:
   - K0: 자동 정리 (허용범위 내)
   - K1: 수동 정리 필요 (초과)
   - K2: 예외 승인 (정책)
   ```

**필드 설명**:

| 필드 | 설명 | 값 | 비고 |
|------|------|-----|------|
| Purchasing Organization | 구매조직 | 1000 | 필수 |
| Tolerance Amount | 금액 상한 | 100000 | KRW |
| Tolerance Percentage | 비율 상한 | 1.0 | % |
| Vendor | 공급업체 | (선택) | 개별 설정 시 |
| Clearing Rule | 정리 규칙 | K0, K1, K2 | 우선순위 |

## GR/IR 계정 구조

### 계정 설정 권장안 (한국식):

```
자산 계정 (99000~99999):
├─ 199000: GR/IR 미정산 (자산쪽 임시)
│   └─ 용도: 상품 입고 후 송장 대기 중
│
├─ 199100: GR 미송장 (재무분석용)
│   └─ 용도: GR/IR 분석 및 보고
│
└─ 199200: 선입선출(FIFO) 조정용
    └─ 용도: 재고평가 조정

부채 계정 (200000~299999):
├─ 200000: 채무
│   └─ 용도: 정상 외상 채무
│
└─ 200100: GR 미정산 채무쪽
    └─ 용도: GR는 들어왔으나 IR 미래
```

## GR/IR 정리 월별 절차

### 1단계: 사전 확인 (월초)

```
T-code: MR2N (GR/IR 조회)
- GR/IR 잔액 확인
- 처리 기한 경과 항목 식별
- 문제 거래 사전 조사
```

### 2단계: 공급업체 협력 (월중)

```
미결제 GR/IR에 대해:
- 공급업체에 송장 지연 확인
- 품질 문제로 미입고 여부 확인
- 송장금액 조정 협상
```

### 3단계: 자동 정리 (월말)

```
T-code: OBYP 또는 자동 배치
- 송장 입력 시 GR/IR 자동 상쇄
- 대부분의 항목 정리됨 (80~95%)
```

### 4단계: 수동 정리 (월말~익월초)

```
T-code: MR11 실행
- 자동 정리 불가 항목 처리
- 허용범위 초과 항목 확인
- 필요시 공급업체에 송장 재발행 의뢰
```

### 5단계: 예외 처리 (필요시)

```
GR/IR 잔액이 지속되면:
1. 구매팀: 공급업체 확인
2. 경리팀: 기간 조정 전기
3. 회계팀: FB03로 잔액 검증

최종 목표: GR/IR 잔액 ≈ 0 (또는 소액)
```

## 일반적 오류 및 해결책

| 오류 | 원인 | 해결책 |
|------|------|--------|
| **GR/IR 잔액 증가** | 송장 입력 누락 | MR2N에서 미정산 확인, 공급업체 추격 |
| **자동 정리 실패** | 허용범위 초과 | OMR6 상한 재검토 또는 공급업체에 송장 조정 의뢰 |
| **이중 정리** | MR11 중복 실행 | 배치 스케줄 확인, 중복 제거 |
| **계정 혼동** | OBYP 설정 오류 | SPRO에서 OBYP 재확인, GL 계정 매핑 검증 |
| **기간 오류** | 차기 기간 GR/IR | 기간 OB52 확인, 선급 입고 거래 검토 |

## 실제 예시: 30일 GR/IR 정리

### 시나리오: 1월 구매 거래

```
2024년 1월 거래:
1. 2024.01.10: 상품 입고 (GR)
   - PO: PO-2024-0001
   - 금액: 5,000,000 원
   - 공급업체: ABC Supplier
   
   자동 전기:
   Debit  530000 (자재비) 5,000,000
   Credit 199000 (GR/IR) 5,000,000

2. 2024.02.05: 송장 수령 (IR) — 26일 경과
   - 송장: INV-202401-001
   - 금액: 5,000,000 원 (정확히 일치)
   
   자동 정리 (OBYP 적용):
   Debit  199000 (GR/IR) 5,000,000 — 선 제거
   Credit 200000 (채무) 5,000,000
   
   결과: 2월 정상 채무 계정

3. 2024.02.28: 월말 GR/IR 확인
   T-code: FS10
   계정 199000 (GR/IR): 0 원 (정리됨)
   계정 200000 (채무): 5,000,000 원 (미지급)
   
   평가: 정상 ✓
```

### 시나리오: GR/IR 초과 (허용범위 이상)

```
2024년 1월 거래:
1. 2024.01.15: 상품 입고 (GR)
   - 금액: 1,000,000 원
   
   Debit  530000 (자재비) 1,000,000
   Credit 199000 (GR/IR) 1,000,000

2. 2024.02.10: 송장 수령 (IR)
   - 금액: 1,050,000 원 (50,000 차이 = 5%)
   - OMR6 허용범위: 1% = 10,000 원
   - 차이: 50,000 원 > 10,000 원 (초과)
   
   자동 정리 불가 → 수동 처리 필요

3. 월말 MR11 수동 정리:
   선택 1) 공급업체 확인 → 송장 정정
   - 송장 재발행: 1,000,000 원 (원래대로)
   - 차액 50,000 원: 별도 청구서
   
   선택 2) 수용 처리
   - Debit  530000 (자재비) 50,000 (추가 비용 인정)
   - Debit  199000 (GR/IR) 1,000,000
   - Credit 200000 (채무) 1,050,000 (전액)
   
   선택 3) 품질 문제 처리
   - Debit  750000 (반품, 운임 등) 50,000 (손실)
   - Debit  199000 (GR/IR) 1,000,000
   - Credit 200000 (채무) 1,050,000
```

## 주의사항

1. **GR/IR 계정 명확성**
   - 임시 계정 199000은 월말 항상 0에 가까워야 함
   - 잔액이 있으면 송장 미수령 의미

2. **OBYP와 OMR6 동시 설정**
   - 둘 다 필수 (OBYP만으로는 불완전)
   - OBYP: 자동 정리 메커니즘
   - OMR6: 허용 범위 제어

3. **공급업체 협력 중요**
   - GR/IR 초과 발생 시 공급업체 먼저 확인
   - 조정 불가 거래는 예외 승인 후 기록

4. **월말 마감 순서**
   ```
   1. MIRO (송장 입력 모든 완료)
   2. OBYP 자동 정리 (배치)
   3. MR11 수동 정리 (필요시)
   4. FS10 / FB03 검증 (GR/IR 잔액 확인)
   5. 기간 잠금 (OB52)
   ```

5. **기간 제어 연계**
   - 2월 기간 개시 전 1월 GR/IR 정리 완료 필수
   - 정리 지연 시 다음 달 미수령 항목과 혼동
