# WM 일간 운영 Best Practice

## 적용 범위
- ECC: ✓ | S/4HANA: EWM 권장 (WM deprecated)

## 개요
창고 관리의 일일 점검 체크리스트. Transfer Order(TO) 처리, 재고 관리, 저장유형 관리를 포함합니다.

---

## 체크리스트

### 미처리 Transfer Order 확인
- [ ] **LT06 미처리 TO 조회** — T-code: LT06
  - 왜: 완성품 출고 지연 방지, 고객 인도 기한 준수
  - 빈도: 매일 08:00, 12:00, 16:00 (3회)
  - 세부:
    ```
    LT06 필터:
    - Status: '생성' 또는 '부분완료' (미완료)
    - 저장유형: 전체 또는 특정 저장유형
    - 우선순위: 1(높음) 필터링
    ```
  - 조치:
    ```
    미처리 TO별:
    - 창고 담당자에게 작업 배정
    - 물리적 위치 확인 (빈 위치)
    - 예상 완료 시간 예측
    ```

### 미확인 Transfer Order 처리
- [ ] **LT06 미확인(Unconfirmed) TO** — T-code: LT06
  - 왜: 재고 정확성, GR(Goods Receipt) 연계 확인
  - 빈도: 매일 17:00
  - 세부:
    ```
    LT06 필터:
    - Status: '미확인'(아직 MIGO나 MMBE로 확인 안 됨)
    - 수령 시간: 어제 이상
    ```
  - 조치:
    ```
    각 TO별:
    - 물리적 확인: 물품이 도착 저장유형에 실제로 있는가?
    - MIGO로 확인 처리 또는
    - 이상 조사 후 LT05로 TO 취소
    ```

### 빈 점유율(Bin Utilization) 분석
- [ ] **LX04 / LX05 저장유형별 점유율** — T-code: LX04
  - 왜: 창고 공간 효율성, 보관 비용 최적화
  - 빈도: 주 1회(금요일 오후)
  - 세부:
    ```
    분석:
    - 각 저장유형별 사용 가능 용량
    - 현재 사용 용량 = (적재 유형 × 수량) / 총 용량
    - 점유율 = 사용 용량 / 사용 가능 용량 × 100%
    
    목표: 70~80% (높을수록 좋지만 과적 방지)
    
    문제:
    - 점유율 < 50%: 불용 저장유형 존재 → 정리 필요
    - 점유율 > 90%: 과적 위험 → 저장유형 추가 또는 재배치
    ```

### WM vs MM 재고 불일치
- [ ] **MMBE vs WM 재고 대사** — T-code: MMBE + LX01
  - 왜: 시스템 재고 vs 물리 재고 오염 방지
  - 빈도: 주 1회(수요일)
  - 세부:
    ```
    비교:
    - MM 모듈(MMBE): 시스템 기준 재고 (총수량)
    - WM 모듈(LX01): 저장유형별 재고 합계
    - 이상: 수량 차이 존재 → 원인 파악
    
    원인:
    - TO 미확인: LT06에서 미확인 TO 정리
    - 이동 오류: 잘못된 저장유형으로 이동
    - 분실: 물리적 손실 → MB1C 조정 이동
    ```

### S/4HANA 마이그레이션 준비 (WM → EWM)
- [ ] **S/4HANA EWM 전환 로드맵 추적**
  - 정책: 
    ```
    ECC WM은 deprecated → S/4HANA로 마이그레이션 필수
    예상 시간: 6~12개월 (프로젝트 규모에 따라)
    ```
  - 준비 단계:
    ```
    Phase 1 (3개월):
    - EWM 이해 교육 (WM 담당자)
    - 현 WM 설정 분석 (저장유형, 전략)
    - EWM 설정 계획 수립
    
    Phase 2 (3개월):
    - EWM Sandbox 환경 구축
    - EWM 설정 수행
    - 마스터 데이터(부품, 저장유형) 마이그레이션
    
    Phase 3 (3개월):
    - User Acceptance Testing (UAT)
    - WM → EWM 병렬 운영(2주)
    - Go-live 실행
    ```
  - 영향: TO 프로세스 변경 최소 (사용 인터페이스는 유사)

## 특수 상황 대응

### 긴급 출고 요청
1. TO 우선순위 변경 (LT06)
2. 물리적 위치 확인
3. 빠른 피킹 수행
4. MIGO 확인 (출고 처리)

### TO 미확인 다량 발생
- 원인: 시스템 오류 또는 GR(입고) 미처리
- 조치: MIGO 일괄 확인 또는 LT05로 취소
