# sap-ewm 한국어 퀵가이드

## 🔑 환경 인테이크

SAP EWM(Enhanced Warehouse Management) 구현/운영 전, 다음을 반드시 확인하세요:

### 1. SAP 플랫폼 및 배포 모드
- **S/4HANA On-Premise**: EWM 2020 이상 권장
- **RISE(Private Cloud)**: EWM 완전 지원 + 자동 업데이트
- **Cloud Public Edition**: EWM 제한 지원 (기본 기능만)
- **한국**: 쿠팡, CJ, 이마트 등 대형 물류센터 기준

### 2. EWM 배포 아키텍처
- **Embedded**: S/4HANA와 동일 인스턴스 (중소규모)
- **Decentralized**: 독립 인스턴스 + RFC 연계 (대규모, 권장)
  - 한국 물류: Decentralized 권장 (처리량 > 5,000건/일)

### 3. 물류센터 규모 및 복잡도
- **규모**: 일일 처리량(입고/출고 건 수)
- **복합성**: 다중 보관 전략(FIFO/LIFO), 크로스 도크, 반품 센터
- **통합**: MM(재고), SD(배송), TM(운송) 연계도

### 4. 한국 특화 요구사항
- **이커머스**: 당일/익일 배송 필수 → 자동 피킹/분류 필수
- **규제**: 개인정보(배송지) 암호화, 배송증 전자화(택배사 연동)
- **운영**: 야간/24시간 운영 → 시스템 안정성 critical

## 📚 핵심 T-code 및 역할

### 모니터링 및 대시보드
| T-code | 기능 | 담당자 |
|--------|------|--------|
| **/SCWM/MON** | EWM 통합 모니터링 대시보드 | 운영 감시자 |
| **/SCWM/ACT** | 활동(Activity) 조회 | 창고팀 |
| **/SCWM/AREA** | 구역(Zone) 및 Bin 현황 | 보관 담당자 |

**/SCWM/MON 대시보드**:
```
- 실시간 처리량: 입고/출고 건수 (시간대별)
- 적체 현황(Bottleneck): 물류 구간별 대기 시간
- 에러/이상: RF 오류, 시스템 오류 건수
- 예측: 당일 예상 처리량 vs 계획
```

### 입고(Inbound) 프로세스
| T-code | 기능 | 담당자 |
|--------|------|--------|
| **/SCWM/GOODS_IN** | 입고 관리 (Goods Receipt) | 입고팀 |
| **/SCWM/PUT_AWAY** | 보관(Put-away) 작업 지시 | 보관팀 |
| **/SCWM/PUTAWAY_MON** | Put-away 모니터링 | 감시자 |

**입고 흐름 (PO 연계)**:
```
1. MM 구매 발주(PO) 생성 → EWM에 자동 전달
2. /SCWM/GOODS_IN: 배송 수령(Inbound Delivery) 등록
3. 상품 스캔 + 품질 검사(QC) → 이상 시 반품 지시
4. /SCWM/PUT_AWAY: 자동 보관 위치 할당
5. 보관팀: RF로 위치 스캔 → 재고 확정
```

### 아웃바운드(Outbound) 프로세스
| T-code | 기능 | 담당자 |
|--------|------|--------|
| **/SCWM/WAVE** | 파도(Wave) 계획 및 실행 | 배송팀/자동화 |
| **/SCWM/PICK** | 피킹(Picking) 작업 | 피킹팀 |
| **/SCWM/PACK** | 패킹(Packing) 및 라벨 | 패킹팀 |
| **/SCWM/SHIP** | 배송(Shipment) 확정 | 배송팀 |

**아웃바운드 흐름 (SO 연계)**:
```
1. SD 판매 오더(SO) 생성 → EWM에 자동 전달
2. /SCWM/WAVE: 당일 오더를 파동(Wave) 단위로 그룹화
   - 파동 크기: 예) 100건 / 파동 또는 시간대별(오전/오후)
3. /SCWM/PICK: 피킹 작업 지시서 출력 (바코드 포함)
4. 피킹팀: RF로 바코드 스캔 → 상품 확인 → 확정
5. /SCWM/PACK: 배송 박스 크기 자동 제안 → 패킹
6. /SCWM/SHIP: 최종 배송 확정 → 택배사 인수증 연동
```

### RF 작업 및 모바일
| T-code | 기능 | 담당자 |
|--------|------|--------|
| **/SCWM/RFUI** | RF 터미널 기본 작업 | 창고팀 |
| **/SCWM/RFUI_WAVE** | RF 피킹 (파도 기반) | 피킹팀 |
| **/SCWM/MOBILE** | 모바일 앱 (Fiori) | 창고팀 |

**RF 작업 예시**:
```
RF 터미널에서:
1. 로그인 → 작업 유형 선택(GOODS_IN / PICK / PACK)
2. 상품 바코드 스캔 또는 위치 입력
3. 시스템: 예상 수량과 비교 → 일치 확인 또는 이상 경고
4. 확정: RF 버튼 → 서버 즉시 반영
```

### 실적 정산 및 인터페이스
| T-code | 기능 | 담당자 |
|--------|------|--------|
| **/SCWM/PI** | Physical Interface(배송증 인터페이스) | 배송팀 |
| **/SCWM/TM_INTERFACE** | 운송관리(TM) 연계 | 운송팀 |
| **/SCWM/CONF** | 배송 확인 및 FI 전기 | 재무팀 |

## 🇰🇷 한국 현장 운영 지침

### 온라인 물류센터(풀필먼트센터) 일상 운영

#### 일일 처리 프로세스 (쿠팡/CJ 규모)
```
오전(06:00~12:00) - 입고 집중
- 공급업체로부터 입고: 500~1,000건
- /SCWM/GOODS_IN + /SCWM/PUT_AWAY 동시 진행
- 목표: 입고 → 보관 위치 할당까지 2시간 이내

정오(12:00~17:00) - 피킹 집중
- /SCWM/WAVE: 당일 배송 오더를 3-4개 파동으로 분할
- /SCWM/PICK + /SCWM/PACK: 병렬 처리
- 목표: 피킹 1시간당 300~500건

저녁(17:00~22:00) - 배송 픽업
- /SCWM/SHIP: 최종 확정
- 택배사(CJ, 한진, DHL) 픽업 → 배송증 인수
- /SCWM/PI: 배송증 데이터 연동 → 고객 추적 가능
```

#### 자동화 시스템 연동
- **분류기(Sorter)**: 자동으로 상품을 배송지별로 분류
  - EWM 연계: /SCWM/PACK에서 sorter 출구 지시
  - 한국: 쿠팡 그로박스(GrobBox), CJ 물류센터(Sorter)
- **피킹 자동화 (AS/RS)**: 자동 창고 시스템
  - EWM 연계: /SCWM/PUT_AWAY에서 자동 Bin 할당
  - 한국: 아직 도입 제한적 (비용), 중형센터부터 확대 중

### 한국 배송/반품 관리

#### 당일/익일 배송 운영
- **파동 계획**: /SCWM/WAVE에서 시간대별(오후 3시 픽업, 밤 8시 픽업 등) 설정
- **배송증 연동**: /SCWM/PI로 택배사 API 자동 연계
  - 택배 추적번호 → 고객 메일/SMS 자동 발송
- **배송 기간**: 주문~배송증 생성까지 < 30분 (경쟁력)

#### 반품 센터 운영
- **반품 입고**: /SCWM/GOODS_IN (반품 구간 별도 Bin)
- **품질 검사(QC)**: 불량 여부 판단
  - 양품: 재출고(재보관)
  - 불량: 폐기 또는 제조사 반송
- **한국 반품율**: 온라인 5-10% → EWM에서 별도 추적 필수

### 바코드/RF/모바일 운영
- **바코드 전략**:
  - 상품 바코드(팔림): EAN-13 (국제)
  - Bin 바코드: 보관 위치 고유 번호
  - 배송 바코드: 배송증 번호 (택배사)
- **RF 터미널**: 
  - WiFi 기반 (LTE 백업)
  - 야간 운영 → 배터리 5-8시간 지속 필요
  - 한국 날씨(습도): 방진/방수(IP65) 필수
- **모바일 앱** (Fiori, SAP Warehouse Management App):
  - 스마트폰으로 피킹/패킹 (RF 터미널 + 모바일 병행)
  - 한국: 점진적으로 모바일 전환 중

### 개인정보 보호 (배송지 관리)
- **한국 규제**: 개인정보보호법 + 배송 정보 보안
- **암호화**: EWM에서 배송지 주소 암호화 저장
- **접근 제어**: 피킹팀은 배송증 번호만 보임, 주소 숨김
- **파기**: 배송 완료 후 90일 경과 시 배송지 삭제

## ⚠️ EWM 구성 선택: Embedded vs Decentralized

### Embedded (S/4HANA 동일 인스턴스)
- **장점**: 구성 간단, 저비용
- **단점**: 시스템 부하 높음, 확장성 제한
- **추천**: 중소 물류센터 (일일 처리량 < 2,000건)

### Decentralized (독립 인스턴스)
- **장점**: 높은 처리량, 시스템 독립성, 확장 용이
- **단점**: 구성 복잡, RFC 연계 관리 필요
- **추천**: 대규모 물류센터 (일일 처리량 > 5,000건)
  - 한국: 쿠팡/CJ/이마트 모두 Decentralized 사용

**한국 참조 아키텍처**:
```
S/4HANA (Core)
    ↓ RFC / oData
EWM Instance (Decentralized)
    ↓ API / EDI
TM Instance (Transport Management)
    ↓ API
Sorter / RF / 택배사 시스템
```

## 흔한 이슈 및 해결

| 증상 | 원인 | T-code 진단 | 해결 방법 |
|-----|------|-----------|----------|
| 피킹 지연 (Wave 적체) | Bin 부족 또는 아이템 배치 오류 | /SCWM/MON | Put-away 전략(FIFO) 최적화 |
| 배송증 미연동 | /SCWM/PI 실패 또는 택배사 API 장애 | /SCWM/PI 로그 | 택배사 API 상태 확인 |
| RF 오류 (Goods Not Found) | RF 스캔 데이터 불일치 | RF 로그 | 상품 바코드 재확인 |
| 재고 불일치 | 입고/출고 미확인 또는 조작 | /SCWM/MON 재고 조사 | Cycle Count 실행 |
| 시스템 성능 저하 | 처리량 > 허용치 | /SCWM/MON 성능 탭 | Wave 크기 조정 또는 리소스 증설 |

## 📊 모니터링 및 KPI

### 실시간 KPI (일간)
- **입고 처리량**: 시간당 입고 건수 (목표: 100~200건/시간)
- **피킹 정확도**: /SCWM/PICK 오류율 (목표: < 0.5%)
- **배송 시간**: 오더 생성~배송증 출력 (목표: < 30분)
- **재고 정확도**: /SCWM/MON vs 실물 (목표: > 99.5%)

### 주간 리뷰
- **시스템 가용도**: EWM 가동시간 (목표: 99.9% SLA)
- **RF 가용도**: 터미널/네트워크 정상 (목표: > 95%)
- **에러 현황**: 시스템/운영 에러 건수 및 원인
