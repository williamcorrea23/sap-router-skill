# Korean Field Language Guide — 한국 SAP 현장체 작성 규약

sapstack 문서·응답·예시는 모두 이 가이드에 따라 작성합니다. "번역체"가 아닌
**현장체**를 기본으로 하되, 공식 용어와의 연결도 반드시 보장합니다.

이 가이드의 모든 구체 용어는 `data/synonyms.yaml`에 정의되어 있고, 이 문서는
**스타일 측면의 원칙**만 다룹니다.

---

## 🎯 네 가지 핵심 원칙

### 원칙 1 — 이중 병기 (Dual Annotation)

한국 SAP 현장에서는 **같은 개념을 세 가지 형태**로 표기합니다:
1. **외래어 발음** (현장 실사용): `코스트 센터`, `페이먼트 메소드`
2. **공식 번역** (감사·매뉴얼·제안서): `원가센터`, `지급방법`
3. **테이블 필드 코드** (기술 스펙): `KOSTL`, `ZWELS`

sapstack 문서 작성 원칙은:

```
첫 등장:   외래어 (공식 번역, 필드코드)
반복 등장: 외래어만
스키마 참조: 필드코드만
대화 출력: 외래어만 (첫 등장에서만 괄호 병기)
```

**✅ 올바른 예**:
```markdown
코스트 센터(원가센터, KOSTL)에 포스팅이 안 들어갔네요. 
이 코스트 센터는 OKEON에서 활성화 확인 필요해요.
```

**❌ 피할 예**:
```markdown
# 번역체만
원가센터에 전기가 되지 않았습니다. 해당 원가센터의 활성화 상태를 확인하십시오.

# 병기 누락
코스트 센터에 포스팅이 안 들어갔네요. (← 공식 용어 모르는 사람이 감사 때 어떻게?)

# 영어만
The posting did not reach KOSTL. (← 한국어 문서의 의미 없음)
```

### 원칙 2 — 발화체 수용 (Conversational Tone)

현장 운영자·엔드유저는 **문어체**가 아닌 **발화체**로 문제를 설명합니다.
sapstack의 symptom 설명과 예시 대화는 반드시 발화체를 기본으로 합니다.

**자주 쓰이는 발화 패턴**:

| 패턴 | 의미 | 예 |
|---|---|---|
| `~ 돌렸는데` | 실행했더니 | "F110 돌렸는데 에러 떠요" |
| `~ 뜨네요` / `~ 떠요` | 화면에 나타남 | "No payment method가 뜨네요" |
| `~ 안 되는데요` | 작동하지 않음 | "포스팅이 안 되는데요" |
| `~ 박아주세요` / `~ 세팅` | 값 입력 | "ZWELS에 T 박아주세요" |
| `~ 꽂았어요` | 값을 넣었다 | "BUKRS에 1000 꽂았어요" |
| `~ 쳤어요` | T-code 실행했다 | "미고 쳤는데 에러" |
| `~ 땄어요` | 생성했다 (특히 TR) | "트포 땄어요" |
| `~ 탔어요` | 처리됐다/진행됐다 | "전표가 탔어요" |
| `~ 튕겨요` | 입력이 거부됨 | "저장이 튕겨요" |
| `~ 잡혔어요` | 걸렸다/탐지됐다 | "신용체크에 잡혔어요" |
| `아시는 분?` | Q&A 요청 | "에러코드 M7 900 원인 아시는 분?" |

**✅ symptom 설명 예**:
```yaml
symptom_ko: "F110 돌리는데 벤더 하나만 No payment method 뜨는 경우"
```

**❌ 피할 예**:
```yaml
symptom_ko: "지급실행 트랜잭션 F110 수행 시 특정 벤더에 대해 유효한 지급방법이 없다는 오류 메시지가 발생하는 상황"
```

### 원칙 3 — 약어 정착성 수용 (Abbreviations)

한국 현장은 **풀네임 없이 약어만으로 소통**하는 경우가 많습니다. 이를
강제로 풀어 쓰지 말고, 첫 등장에서만 괄호로 병기합니다.

**일반 약어** (`synonyms.yaml`의 `abbreviations` 섹션):

```markdown
# 첫 등장
PO(구매발주)를 생성하고 GR(입고)까지 처리했는데 ...

# 이후 반복
PO가 먼저 승인되어야 GR이 가능해요.
```

**모듈별 약어는 절대 풀어 쓰지 않기**:
```
GL · AP · AR · AA · FI · CO · MM · SD · PP · HCM · BTP · BASIS · BC · GTS
```
이들은 **언제나 그대로 씀**. "일반원장 모듈", "매입채무 모듈"로 풀어 쓰면
오히려 현장감이 떨어지고 감사·매뉴얼 톤이 됩니다.

**T-code는 항상 원형 유지**:
```
✅ F110, MIGO, MIRO, ST22, SE16N, XK02
❌ F110 지급실행 트랜잭션, MIGO 입고 트랜잭션
```
T-code 뒤에 설명이 필요하면 **괄호 병기**: `F110(페이먼트런)`, `MIGO(입고)`.
두 번째 등장부터는 T-code만.

### 원칙 4 — 한국 업무 시점 (Business Calendar)

한국 SAP는 마감 주기가 업무의 중심이고, 시점 표현이 매우 구체적입니다.
**"마감 직전"** 같은 애매한 표현 대신 **`D-1`, `월마감 D+3`** 같은
업계 표준 표기를 씁니다.

**표준 시점 표기** (`synonyms.yaml`의 `business_time_expressions`):

| 표기 | 의미 | 사용 예 |
|---|---|---|
| `D-1` | 마감 기준일 하루 전 | "월마감 D-1 점검" |
| `D-Day` / `D-0` | 마감 당일 | "분기마감 D-Day" |
| `D+1`, `D+3`, `D+5` | 마감 N일 후 | "월마감 D+3 확정" |
| `월초` / `월말` | 월 초반/후반 영업일 | "월초 환율 적용" |
| `가결산` | 확정 전 잠정 결산 | "가결산 재고 실사" |
| `확정결산` | 감사 대응 확정 | "확정결산 후 공시" |
| `H1` / `H2` | 상반기/하반기 | "H2 예산 배분" |
| `영업일` | 주말·공휴일 제외 | "D+3 영업일" |

**✅ 올바른 예**:
```markdown
월마감 D-1에 F110 돌렸는데 에러. 마감 D+3까지 해결해야 함.
```

**❌ 피할 예**:
```markdown
월마감 하루 전 지급실행을 수행하던 중 오류가 발생했습니다. 
월마감 완료 예정일까지 3영업일 이내 해결이 필요합니다.
```

## 📐 Mode별 적용 차이

### Mode 1 — Quick Advisory 응답 (단발 질문)

| 요소 | 규칙 |
|---|---|
| 첫 등장 | 외래어(공식번역, 필드) |
| 문장 톤 | 설명형, 간결 |
| 약어 | 첫 등장에서만 풀이 |
| 시점 | 질문 맥락에 맞춰 |

### Mode 2 — Evidence Loop (진단 루프)

| 요소 | 규칙 |
|---|---|
| Turn 1 INTAKE | 완전 발화체 수용 (엔드유저 원문 그대로) |
| Turn 2 HYPOTHESIS | 발화체 + 이중 병기 (운영자가 다른 사람에게 전달 가능해야 함) |
| Turn 3 COLLECT | 체크 설명은 발화체 + 필드코드 |
| Turn 4 VERIFY | Fix/Rollback은 발화체 + 필드코드 + 메뉴경로 |

### 문서(Reference) 작성

| 요소 | 규칙 |
|---|---|
| SKILL.md 본문 | 이중 병기 완전 준수 |
| 코드블록 예시 | 발화체 위주 |
| 스키마 참조 | 필드코드만 |

## 🚫 금기 표현 (Banned Expressions)

sapstack 문서에서 **쓰지 말 것**:

| 금기 | 대체 |
|---|---|
| "전기(轉記)" (한자 풀이) | "포스팅" |
| "~하여 주시기 바랍니다" | "~해주세요" 또는 "~해요" |
| "수행하십시오" | "돌려주세요" / "쳐주세요" |
| "오류가 발생합니다" | "에러 떠요" / "안 돼요" |
| "확인을 부탁드립니다" | "확인해주세요" / "봐주세요" |
| "본 시스템에서" | "여기서" / "이 시스템에서" |
| "해당 트랜잭션" | "이 T-code" |
| "오퍼레이션" | "작업" / "처리" |
| "인풋" / "아웃풋" | "입력" / "결과" |
| "리소스" (IT 외 맥락) | "자원" / "사람" |

## 🎓 체크리스트: 문서를 쓸 때 스스로 점검

새 SKILL.md, references, 커맨드 문서를 쓸 때 **10초 자가 점검**:

- [ ] 첫 등장에 외래어(공식번역, 필드코드) 이중 병기가 있나?
- [ ] 예시 대화는 발화체인가? ("돌렸는데", "뜨네요" 등)
- [ ] 약어(TR, PO, GR 등)를 불필요하게 풀어 쓰지 않았나?
- [ ] T-code는 원형 그대로 유지했나? (`F110`, not "F110 트랜잭션")
- [ ] 시점은 `D-1`, `월마감 D+3` 같은 업계 표준 표기인가?
- [ ] "~하여 주시기 바랍니다" 같은 존대 매뉴얼체가 섞이지 않았나?
- [ ] 엔드유저 원문은 수정 없이 그대로 보존했나?

7개 중 5개 이상 걸리면 다시 쓰기.

## 🌍 다국어에서의 적용

같은 원칙은 다른 언어에도 변형 적용됩니다:

### 🇩🇪 독일어
- Dual annotation: `Kostenstelle (KOSTL)`
- 현장체는 독일 내부 관행 따르기 (e.g., "Beleg durchgeknallt" = "전표 튕겼다")

### 🇯🇵 일본어
- Dual annotation: `原価センタ (KOSTL)`
- カタカナ 표기도 병행 (`コストセンタ`)
- 敬語는 사용하지 않음 (매뉴얼 톤 회피)

### 🇺🇸 영어
- Dual annotation: `Cost Center (KOSTL)`
- 이미 네이티브 영어라 발화체 수용만 신경 (e.g., "Payment run bombed" = 장애 발생)

## 🔁 AI가 응답 생성할 때 자동 준수

Evidence Loop 수행 중 AI는 다음을 자동으로 적용해야 합니다:

1. **`data/synonyms.yaml` 로드** → 쿼리 토큰을 canonical로 정규화
2. **운영자 원문 보존** → `initial_symptom.description`은 수정 금지
3. **첫 등장 시 이중 병기** → turn-formats.md의 출력 포맷 참조
4. **반복 시 외래어만** → 같은 응답 내에서 재등장은 짧게
5. **T-code·필드코드 유지** → `XK02`, `ZWELS`는 그대로
6. **업무 시점 표기 유지** → `D-1`, `월마감 D+3`은 그대로

이 자동 적용이 실패하면 `audit_trail`에 `style_violation`으로 기록합니다
(v1.6.0 예정).
