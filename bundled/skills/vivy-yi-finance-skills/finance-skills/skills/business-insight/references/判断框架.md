# 判断框架 — 商业洞察异常处理

## 核心原则

**商业洞察的目标是"洞察真实、建议可执行、业务能落地"。**
遇到异常时，优先判断是"数据问题"还是"分析逻辑问题"，确保洞察真正驱动业务改进。

---

## 异常类型与处理方式

### 类型一：分析结论与业务实际严重不符

**典型场景：**
- 分析建议降本但业务反映成本已到极限
- 建议开拓市场但实际竞争格局不支持
- KPI 改善建议与业务优先级冲突

**判断流程：**

```
第一步：确认问题

□ 洞察结论：[描述]
□ 业务反馈：[描述]
□ 冲突类型：
□ 数据失真
□ 分析假设错误
□ 业务理解偏差
□ 优先级认知差异

□ 冲突程度：
□ 局部（单个洞察点）
□ 整体（分析框架有问题）

第二步：分析根因

□ 数据问题：
□ 数据来源：[ERP/BI/业务系统]
□ 数据时间：[是否最新]
□ 数据完整性：[X%]

□ 分析假设：
□ 假设1：[描述] — 是否成立 [✅/⚠️]
□ 假设2：[描述] — 是否成立 [✅/⚠️]

□ 业务理解：
□ 对业务流程理解：[✅ 准确 / ⚠️ 部分偏差 / 🔴 根本偏差]
□ 对竞争格局理解：[✅ 准确 / ⚠️ 部分偏差 / 🔴 根本偏差]

第三步：修正与迭代

□ 修正分析：
□ 数据重新提取：[✅ 已完成 / ⚠️ 待完成]
□ 假设重新验证：[✅ 已完成 / ⚠️ 待完成]
□ 业务语言重新翻译：[✅ 已完成 / ⚠️ 待完成]

□ 新洞察产出：
□ 修正后结论：[描述]
□ 置信度：[✅ 高 / ⚠️ 中 / 🔴 低]
□ 可执行性：[✅ 高（已有具体行动计划）/ ⚠️ 中 / 🔴 低]

□ 与业务对齐：
□ 业务负责人确认：[✅ 是 / ⚠️ 否]
□ 确认人：[姓名]
```

### 类型二：洞察触发条件误报

**典型场景：**
- 预算偏差 > [X]% 触发分析但实际是季节性波动
- 异常值被识别为风险但实际是数据错误
- 预警触发后业务已在上报处理

**判断流程：**

```
第一步：确认误报

□ 触发洞察的条件：[描述]
□ 触发原因分析：
□ 真实异常：[是/否]
□ 季节性波动：[是/否]
□ 数据异常：[是/否]
□ 业务已知并在处理：[是/否]

□ 误报类型：
□ 阈值设置过低（正常波动触发）
□ 数据源异常（数据错误触发）
□ 业务已处理（重复触发）

□ 误报频率：[X] 次/月

第二步：调整机制

□ 触发条件优化：
□ 原阈值：[X]%
□ 建议阈值：[X]%
□ 调整理由：[描述]

□ 过滤机制增加：
□ 增加季节性调整：[✅/⚠️]
□ 增加数据质量检查：[✅/⚠️]
□ 增加业务已知检查：[✅/⚠️]

□ 调整实施：
□ 调整系统：[ERP/BI/分析平台]
□ 调整时间：[YYYY-MM-DD]
□ 调整验证：[✅ 通过 / ⚠️ 待测]
```

### 类型三：专题洞察请求无法满足

**典型场景：**
- 业务请求深度专题分析但数据不足
- 须跨多个系统数据但系统未打通
- 须外部数据但无法获取

**判断流程：**

```
第一步：评估请求

□ 洞察请求：[描述]
□ 请求方：[业务负责人]
□ 请求时间：[YYYY-MM-DD]
□ 截止时间：[YYYY-MM-DD]

□ 数据需求：
□ 内部数据需求：[描述] — 可获得 [✅/⚠️]
□ 外部数据需求：[描述] — 可获得 [✅/⚠️]

□ 可行性评估：
□ 可完全满足：[是 → 执行完整分析]
□ 可部分满足：[是 → 明确范围再执行]
□ 无法满足：[否 → 协调资源或降低预期]

第二步：资源协调

□ 内部资源：
□ 数据工程师支持：[✅ 可获得 / ⚠️ 需申请]
□ 业务分析师支持：[✅ 可获得 / ⚠️ 需申请]
□ 预计完成时间：[YYYY-MM-DD]

□ 外部资源：
□ 外部数据采购：[✅ 可获得 / ⚠️ 成本过高]
□ 咨询公司支持：[✅ 可获得 / ⚠️ 成本过高]

□ 时间与质量权衡：
□ 按时完成但数据有限：[✅ 可接受]
□ 等待完整数据但超时：[⚠️ 业务沟通新截止日期]
□ [X] 天后完整交付：[✅/⚠️]
```

## 升级触发条件

- 重大业务风险洞察须立即升级 CFO
- 分析结论与管理层判断严重冲突须 CEO 裁决
- 洞察方法论存在根本性问题须升级 CFO 审批新的分析框架
