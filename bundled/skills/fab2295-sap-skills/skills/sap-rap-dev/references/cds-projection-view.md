# CDS projection view

A **projection view** is the layer of a RAP BO that's exposed to a service.
It sits on top of the interface view (`I_…`) and gives the service its
shape — fields, UI annotations, redirected compositions, computed columns.

```
Service binding   →  OData V4 / V2 / REST / SQL
Service definition →  expose R_Travel as Travel
Projection view    →  define root view entity R_Travel as projection on I_Travel  ← THIS FILE
Interface view     →  define root view entity I_Travel as select from ztravel
DDIC / table       →  ztravel
```

> **Anchored to**:
> https://help.sap.com/docs/abap-cloud/abap-cds-development-user/projection-view
> https://help.sap.com/docs/abap-cloud/abap-rap/projection-behavior-definitions

## 1. Syntax

```abap
@EndUserText.label: 'Travel — projection for UI'
@AccessControl.authorizationCheck: #CHECK
@Metadata.allowExtensions: true
@Search.searchable: true
define root view entity R_Travel
  provider contract transactional_query
  as projection on I_Travel
{
  key TravelUUID,
      @Search.defaultSearchElement: true
      @Search.fuzzinessThreshold: 0.8
      TravelID,
      AgencyID,
      _Agency.Name        as AgencyName,
      CustomerID,
      _Customer.LastName  as CustomerName,
      BeginDate,
      EndDate,
      BookingFee,
      TotalPrice,
      CurrencyCode,
      OverallStatus,
      Description,
      LastChangedAt,

      /* redirected composition (parent → projected child) */
      _Booking : redirected to composition child R_Booking,

      /* associations to other BOs / reuse entities */
      _Agency  : redirected to R_Agency,
      _Customer
}
```

Building blocks:

| Element                                     | Meaning                                                       |
|---------------------------------------------|---------------------------------------------------------------|
| `define root view entity` / `define view entity` | Same root vs. child distinction as interface views.       |
| `provider contract <kind>`                  | Declares which service kind consumes this projection.         |
| `as projection on <interface_view>`         | Source — must be a view entity, not a table.                  |
| `_Child : redirected to composition child <ProjectedChild>` | Required when the parent is projected and the child is projected too. |
| `_Assoc : redirected to <ProjectedTarget>`  | Optional — re-points an association at its projection. |

---

## 2. Provider contracts

Each projection declares the service shape it supports:

| Contract                       | Purpose                                                          |
|--------------------------------|------------------------------------------------------------------|
| `transactional_query`          | Transactional UI/API view — read + write through RAP.            |
| `transactional_interface`      | Transactional API view — exposed via OData Web API binding.      |
| `analytical_query`             | Analytical query for a multi-dimensional consumer.               |
| `analytical_dimension`         | Dimension in an analytical model.                                |
| `analytical_cube`              | Cube in an analytical model.                                     |
| `analytical_kpi`               | KPI in the analytical model.                                     |
| `service_consumption`          | Generated by the Service Consumption Model — don't author by hand. |
| `extension_view`               | Used inside view extension scenarios.                            |

For a RAP BO consumed by a Fiori app, use **`transactional_query`** on the
projection that backs the OData V4 UI binding.

## 3. Redirecting compositions and associations

When a parent projection (`R_Travel`) wants to expose its child composition
through the projected child (`R_Booking`), it must **redirect**:

```abap
_Booking : redirected to composition child R_Booking,
```

Without the redirect, the composition still points at the interface child
`I_Booking`, which doesn't belong to the service.

Associations to other projection views can also be redirected:

```abap
_Agency : redirected to R_Agency,
```

When omitted, the association resolves to whatever target the interface view
declared (often an `I_…` view, which is fine if it's also exposed by the
service or by a different reachable contract).

## 4. Adding fields beyond the interface

A projection can:

- Hide fields by simply not listing them.
- Reshape (rename) fields with `<source> as <alias>`.
- Surface associated fields: `_Agency.Name as AgencyName`.
- Add virtual elements (computed in ABAP via an exit class) — see
  [cds-view-entity.md §5.2](cds-view-entity.md#52-virtual-element-computed-in-abap-via-exit-class).

A projection **cannot** add brand-new persistent fields — those have to be
on the interface view (and ultimately on the table).

## 5. UI annotations on the projection

`@UI.*` annotations live on the projection (not the interface) so different
projections of the same interface can have different UI shapes.

```abap
@UI: {
  headerInfo: { typeName: 'Travel', typeNamePlural: 'Travels',
                title: { value: 'TravelID' },
                description: { value: 'Description' } }
}
define root view entity R_Travel
  provider contract transactional_query
  as projection on I_Travel
{
  key TravelUUID,

      @UI: { lineItem:      [ { position: 10, label: 'Travel ID' } ],
             identification:[ { position: 10, label: 'Travel ID' } ],
             selectionField:[ { position: 10 } ] }
      TravelID,

      @UI: { lineItem:      [ { position: 20, label: 'Agency' } ],
             identification:[ { position: 20, label: 'Agency' } ] }
      AgencyID,

      @UI: { lineItem:      [ { position: 30, label: 'Status',
                                criticality: 'OverallStatusCriticality' } ],
             identification:[ { position: 30, label: 'Status',
                                criticality: 'OverallStatusCriticality' } ] }
      OverallStatus,

      @UI.hidden: true
      virtual OverallStatusCriticality : abap.int1,

      LastChangedAt
}
```

The full annotation catalog is in [cds-annotations.md](cds-annotations.md).

## 6. Projection behavior definition

A projection view always needs its own BDEF (alongside the interface
BDEF). The projection BDEF *delegates* to the interface BDEF via `use`.

```abap
projection;
strict ( 2 );
use draft;

define behavior for R_Travel alias Travel
{
  use create;
  use update;
  use delete;

  use association _Booking { create; with draft; }
  use association _Agency;
  use association _Customer;

  use action acceptTravel;
  use action rejectTravel;
  use action Activate;
  use action Discard;
  use action Edit;
  use action Resume;
  use action Prepare;
}
```

What can appear in a projection BDEF:

- `use create | update | delete` — exposes the matching CRUD operation.
- `use association <name> { … }` — exposes a child / sibling association.
- `use action <action>` / `use function <fn>` — exposes a typed operation.
- `use action <DraftAction>` — exposes the draft actions (Activate, Edit,
  Discard, Resume, Prepare).
- Field-level `field ( read only )` etc. — to *further restrict* fields
  beyond what the interface allows. A projection can **only restrict**, not
  relax.

See [behavior-definition.md](behavior-definition.md) for the full BDEF
language reference.

## 7. Multiple projections of the same interface

Common pattern: an interface BO surfaces multiple projections — a UI one and
an API one — each with its own service definition and binding.

```
I_Travel (interface)
   ├─ R_Travel_UI    (transactional_query)  → ZUI_TRAVEL    (OData V4 UI)
   └─ R_Travel_API   (transactional_interface) → ZAPI_TRAVEL (OData V4 Web API)
```

The UI projection has full `@UI.*` annotations; the API projection has none
but might suppress draft (`use create; use update; use delete;` without
`use draft;`) for direct create scenarios.

---

## 8. Common gotchas

- ❌ Adding a field on the projection that doesn't exist on the interface —
  the projection compiles only fields it can resolve.
- ❌ Forgetting `redirected to composition child` — the projected child is
  unreachable through the projected parent.
- ❌ Mixing `provider contract analytical_query` with a transactional BDEF —
  the contract and the BDEF kind must match.
- ❌ Trying to add new operations on the projection BDEF — `use` exposes
  what the interface defines; truly new behavior belongs on the interface
  BDEF (or as an extension on a released base BDEF).
