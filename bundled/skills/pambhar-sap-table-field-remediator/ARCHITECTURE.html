SAP Table & Field Remediator

Architecture & the Cloudification Repository integration

Claude Code plugin ECC → S/4HANA brownfield Deterministic + LLM residue branch: dogukan_cloudificationrepo

This tool scans a company's custom ABAP code and flags every database access that will break when they upgrade from the old SAP ECC system to S/4HANA. It rates each issue by how much human judgment the fix needs, and writes a machine-readable report. Detection is deterministic; the LLM only touches the hard residue; a human signs off.

Moving-house analogy. You have a notebook of instructions like "spare keys are in the blue room's top drawer." You move house — some rooms are renamed, some furniture is thrown out. This tool reads your notebook and highlights every instruction that won't work in the new house, before you move.

1. Vocabulary (3 words)

TermPlain meaningExample
Tablea big spreadsheet inside SAPBSEG = accounting line items
Fielda column in that spreadsheetHKONT = account number
Successor / CDS viewthe new place to read the same data after the upgradeBKPF → I_JournalEntry

A CDS view is SAP's modern "ready-made counter" layer on top of raw tables — a pre-built query that joins tables, fixes field names, and is a stable contract. S/4HANA wants new code to read the CDS view, not the raw table.

Decision vs. target — the key split. The decision ("does it break? how hard?") always comes from the Catalog. CRV never touches world/tier — it only helps with the target ("what do I use instead?").

4. How the Cloudification Repository was added

Four parts: source → builder → data → loader, then two wiring points in classify.py.

Build time (run rarely, by hand)

# 1. download SAP's full open dataset (Apache-2.0, 44,541 objects)
curl -sSLo objectReleaseInfo_PCELatest.json \
  https://raw.githubusercontent.com/SAP/abap-atc-cr-cv-s4hc/main/src/objectReleaseInfo_PCELatest.json

# 2. distil it into a small target dictionary (262 tables with a successor)
python3 mcp/build/crv_ingest.py --crv objectReleaseInfo_PCELatest.json \
  --out skills/.../references/crv-successors.json --types TABL

The builder does three things: filter (keep only TABL that are notToBeReleased and have a successor), pick the best successor (preference order CDS_STOB > TABL > FUNC > CLAS > INTF), and write the JSON. Result: 44,541 → 2,832 tables → 262 with targets.

5. The logic — two flows

Everything hinges on one question: is the table in the Catalog? CRV behaves differently in each branch — but in both it only ever concerns the target, never the decision.

Flow A Table IS in the catalog

A finding is produced. Decision & tier come from the catalog. CRV is always consulted here (not "only on a miss") — to check the target:

  • CRV empty → leave the catalog target as-is
  • CRV = catalog → note "confirmed by CRV"
  • CRV ≠ catalog → note "CRV lists X; confirm" (keep catalog, flag both)
  • catalog empty, CRV set → fill from CRV

Example — BSEG. Catalog says I_JournalEntryItem; CRV says I_OperationalAcctgDocItem → different → the report keeps ours and adds "CRV lists I_OperationalAcctgDocItem; confirm."

Flow B Table is NOT in the catalog

No finding — it goes to the review-queue (unknown ≠ safe). This is the "after a catalog miss" case. CRV attaches a suggested target so a reviewer doesn't start from a blank.

{
  "object": "MARD",
  "reason": "not_in_catalog",
  "crv_successor": "I_ProductStorageLocationBasic",
  "crv_successor_type": "CDS_STOB",
  "crv_state": "notToBeReleased"
}

The verified item stays in the queue until a human says fix / skip. It never auto-enters the scored report.

In one line. A in catalog → CRV verifies / completes the target. B not in catalog → CRV suggests a target. In both, the decision (does it break / tier) is always the Catalog — CRV never overrides it. And what CRV verifies is the successor (what to use instead), not the table name itself.

6. Why CRV is a target dictionary, not a break-list

What the CRV state values mean

These are clean-core / ABAP Cloud API-governance labels — "are you allowed to use this object in released ABAP Cloud code?" — not "does it still physically exist / work after a brownfield conversion?"

StateMeaning
released SAP published this as an official, stable public API — usable in ABAP Cloud / clean-core code. Usually a CDS view (e.g. I_JournalEntry).
notToBeReleased SAP will not release it as an API — use its successor in clean core. Not the same as "removed": most raw tables (BKPF, MARA) still exist and a classic SELECT still works after a brownfield conversion; SAP just steers new code to the successor.
notToBeReleasedStable Same as notToBeReleased, but not expected to change soon (stable).
deprecated On its way out of use.

Analogy. released = "on the shelf, warranted — use it." notToBeReleased = "we no longer sell this; buy the newer model" — the old one may still work at home, but the store won't recommend it for new work.

Two different questions

CRV's state answers a different question than ours. Importing it as a break-list would wreck precision.

SourceQuestion it answers
Our CatalogDoes this table break in an ECC → S/4HANA brownfield conversion?
CRV stateIs this a released, clean-core API for ABAP Cloud?
TableCRV stateReads fine in brownfield?Our catalog
MARA, VBAK, BKPF, T001notToBeReleasedyes — worksVALID (suppress)
BSEGnotToBeReleased → I_OperationalAcctgDocItemruns, but data movedmust-fix (restructured)
VBUK, RFBLG, LISabsent from CRVno — empty/removedmust-fix (abolished)

Proof it can't be the break-list. MARA, VBAK and BKPF are all notToBeReleased yet still read fine — trusting CRV as a break-list would raise false alarms. Note BKPF (document header) and BSEG (line items) share the same CRV state, but our catalog rates BKPF VALID and BSEG must-fix — CRV can't tell them apart; only the catalog can. And the tables that truly break (VBUK) are absent from CRV, because a deleted table can't be a "non-released API."

Verified safe. After the integration, MARA is still correctly suppressed, and findings/tiers are identical to the pre-CRV run. CRV is purely additive — remove the dictionary and behaviour is unchanged.

7. Design boundaries (locked in)

  1. Classify only. detect / guard / analyze / the report schema are untouched. Minimal surface.
  2. Optional. No dictionary → {} → catalog-only behaviour. Zero breakage risk.
  3. Catalog wins. CRV never sets world/tier — only replacement and review-queue hints.
  4. Layered by construction. data (json) / builder (ingest) / loader (crv.py) are separate; refreshing SAP data is just re-running the builder.

8. What it does NOT do (yet)

  • No ready-to-paste ABAP. Findings carry a target name + guidance + an intent question — not a rewritten SELECT. The patch field is always null; apply-mode is roadmap.
  • Tables only. The dictionary currently keeps TABL (262). Views / data elements could be added via --types.
  • No field-level in CRV. Renames like HKONT → RACCT and the difficulty tier stay in the hand catalog — CRV has neither.
