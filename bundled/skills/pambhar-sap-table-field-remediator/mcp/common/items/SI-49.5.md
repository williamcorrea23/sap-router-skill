## Symptom

You plan to use self-billing with your buyer. The following SAP S/4HANA Transition Worklist item is applicable in this case.

## Reason and Prerequisites

Self-Billing (SD-SBI-INV) is not strategic, and customers should evaluate the new industry cloud solution SAP Self-Billing Cockpit

Self-Billing (SD-SBI-INV) is a deprecated component. This means that it is still supported but will be replaced by the SAP Self-Billing Cockpit in the long term. The date is not set yet.

## Solution

SAP Self-Billing Cockpit (SD-BIL-SBI) is an industry cloud solution helping suppliers to efficiently process self-billing documents transmitted from the buyer and prepares the documents for the clearing process in accounts receivable.

It enhances efficiency through automated simulation result and combined with SAP Fiori and monitoring capabilities help end users take faster decisions.

SAP Self-Billing Cockpit also supports both Self Billing with Automatic posting (so called 'SBWAP' processing) and Self-billing with Invoice creation (so called 'SBINV' processing)

Being an Industry cloud solution SAP Self-Billing Cockpit is aligned with SAP's strategy and has continuous investment with a well-defined roadmap

SAP Self Billing Cockpit integrates with S/4 HANA back-end systems S/4HANA Cloud and S/4HANA OP. For prerequisites, please refer to chapter 'Relevancy'.

## Business Process related information

## Self-Billing with Automatic Posting ('SBWAP')

You can only use self-billing with automatic posting for scenarios in which exactly one invoice is created for each outbound delivery. To be specific, no convergence or split of billing data from deliveries can occur during invoice creation.

The following processing flows exist in self-billing with automatic posting:

## Standard processing involving value and quantity

This is the standard type of processing, where the supplier creates billing documents for outbound deliveries. After receiving self-billing documents from the buyer, the system determines the corresponding reference delivery document, and compares the received self-billing documents with the corresponding reference billing documents.

## Value-based clearing invoice without retroactive billing

This type of processing assumes that the transmission explicitly states that it is a clearing invoice, by referring to a transaction that has already been settled. In addition, such transmissions deal exclusively with value changes (for example, retroactive price changes) claimed by the customer (that is, the buyer). Therefore, the received quantity is not considered during value comparison or further processing.

## Value-based clearing invoice with retroactive billing

This type of processing also deals exclusively with value changes (for example, retroactive price changes) claimed by the customer (that is, the buyer), by referring to a transaction that has already been processed and settled.

Self-Billing with Invoice creation ('SBINV')

In some countries or regions, legal requirements dictate that the billing documents that the supplier creates as part of a self-billing process must be identical with the self-billing documents that SAP Self-Billing Cockpit receives in transmissions from the buyer. In this scenario, known as self-billing with invoice creation, the supplier can create the final billing documents only after they have received self-billing documents from the buyer. In self-billing with invoice creation, the buyer can send several self-billing documents for one delivery item. SAP Self-Billing Cockpit continues to create billing documents as the buyer sends them, until the delivery quantity is fully billed.

The following processing flows exist for self-billing with invoice creation:

## Initial invoice creation

This type of processing deals with initial invoice creation. After receiving self-billing documents from the buyer, the system determines the corresponding delivery document for reference, and compares the received data with the reference data.

Based on the comparison results, the system proceeds in one of the following ways:

It executes self-billing to trigger the creation of invoices (billing type CIXS) for billing items where there is no difference.

It executes self-billing to create a clearing line, in addition to an invoice, for billing items where the transmitted net value differs from the reference net value.

It executes self-billing to create a clearing line and an open line, in addition to an invoice, if the difference exceeds the predefined tolerance limits.

## Credit or debit memos for quantity corrections

This type of processing deals with quantity corrections (such as damaged goods during the transport) claimed by the buyer, by referring to a transaction that has already been processed and settled. Therefore, the buyer sends a credit or debit memo for quantity corrections with reference to one or more existing invoices.

During the process, the system compares the claimed net value change against the internal net value change, both of which are calculated based on input such as quantity changes of the settled transaction.

If they match, the system executes self-billing to trigger the creation of a credit memo (billing type CIXG) or debit memo (billing type CIXL) exactly as the buyer has prepared it.

If they don't match, the system creates a credit or debit memo and a clearing line, as well as an open line if the difference exceeds the predefined tolerance limit.

## Credit or debit memos for retroactive price changes

This type of processing is like the previous one except that it deals with retroactive price changes. Therefore, the system compares the claimed net value change against the internal retroactive billing value, which is calculated in your system based on input such as price changes of the settled transaction.
