## Symptom

You are doing a system conversion to SAP S/4HANA. The following SAP S/4HANA Transition Worklist item is applicable in this case.

## Reason and Prerequisites

The Engineering Change Request / Engineering Change Order functionality is no more available with SAP S/4HANA.

## Solution

## Description

With SAP S/4HANA the current the Engineering Change Request / Engineering Change Order functionality within SAP Product Lifecycle Management (SAP PLM) is not available anymore. It is recommended for new customers to use Change Record instead for tracking, approving and facilitating (change management) purposes. Alternatively Engineering Record (only available on Web UI as part of the compatibility package) could be assessed.

## Business Process related information

Managing (tracking, approving and facilitating) changes will still be possible. The Change Record / Engineering Record shall be used for this.

The change number will serve to identify the explicit changes and the Change Record / Engineering Record is to be used to manage the whole process. The approval process, the objects to be changed will all be part of the Engineering Record, while the change number will solely be used to facilitate the changes after approval has been given.

Transaction not available in SAP S/4HANA on-premise edition 1511

CC31 - Create Change Request

CC32 - Change Change Request

CC33 - Display Change Request

CC62 - Conversion of change packages

OS72 - ECH: Workflow for ECR/ECO

OS73 - ECH: Workflow for Object Mgmt Record

## Required and Recommended Action(s)

Existing and running Change Requests and Change Orders should be finished before system upgrade is facilitated as working with these type of object will not be possible in S/4HANA. Additional settings will be required in the system for the Customer to use the Change Record / Engineering Record. Set up Change Record / Engineering Record (Types, status network, BRF+) and Process Route (Activities, services, process route templates, BRF+) for tracking, approving and facilitating business object related changes.

## How to Determine Relevancy

The Engineering Change Request / Engineering Change Order functionality is used if entries exist in DB Table AENR where the CCART field is fileld (check with SE16)

Use ST03N (on the production system, if possible) to check transaction profile of the complete last month, whether the following transactions have been used: CC31, CC32, CC33, CC62, OS72, OS73

## Other Terms

Change Number, Change Master, Change Management, Engineering Change Request, Engineergin Change Order, ECR, ECO

## Modernized topics

The Engineering Workbench transaction (CEWB) has been modernized as part of SAP S/4HANA 2023 starting with FPS03 release, or later. The modernization incorporates the harmonized use of standardized SAP S/4HANA data access methods. The modernized transaction is part of the perpetual scope of SAP S/4HANA. ## 28.24S4TWL - Behavior change for AVC Sales Order (SET) scenario configuration models

## Application Components:LO-VCH

Related Notes:

| Note Type       |   Note Number | Note Description                                                                |
|-----------------|---------------|---------------------------------------------------------------------------------|
| Business Impact |       3190519 | S4TWL - Behavior change for AVC Sales Order (SET) scenario configuration models |

## Symptom

## Business Value

In the future it shall be possible to safely reuse planned order / production order configuration models in bundles (configuration models with sales order (SET) configuration profiles) in advanced variant configuration (AVC).

## Description

The safe reuse is enabled by introducing a form of encapsulation for object dependencies from planned order / production order configuration models if they are used in bundles / SETs. For a detailed explanation please see below.

## Reason and Prerequisites

A planned order / production order configuration profile (PPO profile) is used for standard configuration models. Typically, configurable materials with this type of profile are configured and sold standalone. But it is also possible to bundle several such PPO models and to sell them together as a set. Technically, this is achieved by creating a configuration model with profile type sales order (SET) scenario and by using the PPO models as the leafs in the BOM.

Example If you want to sell a bundle consisting of a configurable laptop and some configurable accessories you create a sales order (SET) scenario configuration model where the laptop and each of the accessories are a leaf item in the maxBOM of the bundle. Only the laptop and the accessories are produced, not the SET (requirements generation at component level).

Presently, the AVC supports only PPO and SET profiles as configurable components in bundles / SETs.

This reuse of PPO models in bundles, however, causes problems in the interpretation of object dependencies. More precisely, there are two distinct issues:

If the PPO model contains an object dependency that refers to $parent / $root: Imagine, for example, that characteristic X shall take its value from characteristic Y in the PPO configuration model. This is frequently expressed via procedures of the form $self.X = $root.Y , where the assignment target is $self and the "read"-characteristic is on $root. In a PPO model without BOM explosion $self and $root refer to the same object and everything works as expected. But if the PPO model is configured as part of a bundle $root suddenly refers to the root item of the bundle model - the object dependency has changed its meaning. Please note that this phenomenon is not limited to procedures or $root. It affects all types of object dependencies and all uses of $parent and $root in the PPO configuration model (keep in mind that PPO models with BOM explosion are not supported in bundles yet).

If the PPO model contains a constraint that relates characteristics from different classes, and those classes occur more than once in the bundle: Imagine that characteristic X from class CX shall be equal to characteristic Y from class CY (and vice versa). This is frequently formulated as a constraint

OBJECTS: CX IS_A (300) CX, CY IS_A (300) CY.

RESTRICTION: CX.X = CY.Y.

If the PPO model is now included in the bundle twice, the following issue occurs: If you manually set a value for X on the one instance of the PPO model, the constraint automatically assigns Y, but it does so on both the one and the other instance. This happens because constraints match globally, everywhere where the classes addressed occur. However, in our example, clearly this is an unwanted side-effect from one instance of the PPO model into the other. Please note that this phenomenon is not limited to including the same PPO model more than once into the bundle - it occurs everytime when different items in the bundle make use of the same classes addressed by a constraint on one of the items.

These issues are present in AVC's support of the sales order (SET) scenario today, just as they are present in LO-VC's support of the scenario. For the AVC we have decided to change this situation by introducing encapsulation for object dependencies for the reuse of PPO models in bundles. The new behavior can be described as follows:

Object dependencies belonging to a PPO model in a bundle affect only the PPO model they belong to.

Please note that it is still possible to express relations between the different items in a bundle:

Object dependencies belonging to the SET model affect all the items in the bundle / have full visibility.

Solution

## How to Determine Relevancy

This note is relevant for you if you have (or are about to create) sales order (SET) configuration profiles for advanced variant configuration. This is the case if in table CUCO there is an entry with KDSET = X and PROC_MODE = A.

## Transitioning

The new behavior does not become effective immediately. For a period of one S/4 release both behaviors are available so that remodeling is possible. Once you have adjusted your configuration models to the new behavior you can switch on the new behavior of AVC as follows:

In S/4 OP go to transaction VCH_NEW_BUNDLING to switch on the new behavior.

In S/4 Cloud enable toggle IDEA_CONFIGURE_NEW_BUNDLING in the feature management cockpit.

The new behavior becomes mandatory with the 2023 OP / the 2302 Cloud release of S/4 HANA.

## Required and Recommended Actions

Please check your sales order (SET) scenario configuration models and adjust them as needed. We recommend you pay particular attention to the following points:

If today you relate characteristics from the bundle with characteristics from the PPO model using $self, $parent and $root, the following idiom may be helpful for replacing the dependencies. Imagine you have a procedure for value inheritance on the PPO model that reads $self.X = $root.Y where (different from our previous example) X belongs to the PPO model, and Y to the bundle. Furthermore assume that $self refers to the instance of class C_COMP that is on position '0020' of the BOM of $root. Then the following constraint achieves a similar value inheritance:

OBJECTS: COMP IS_A (300) C_COMP, BUND IS_A (300) C_BUND.

CONDITION: PART_OF(COMP,BUND,'0020').

RESTRICTION: COMP.X = BUND.Y.

With this pattern, the value assignment works from top to bottom and vice versa. If you want inheritance to be only from top to bottom, use the following constraint pattern:

OBJECTS: COMP IS_A (300) C_COMP, BUND IS_A (300) C_BUND.

CONDITION: PART_OF(COMP,BUND,'0020').

## RESTRICTION: COMP.X = BUND.Y IF SPECIFIED BUND.Y.

Note that using these idioms you get a constraint assignment where previously there was a procedure assignment. There may be situations where you want to keep authorship, for example, when inheriting a value as a default. Imagine that our previous example is changed to set a PPO model default from a bundle-level characteristic via a procedure $self.X ?= $root.Y on the PPO model. Here we recommend using a helper characteristic XHelper together with a constraint:

OBJECTS: COMP IS_A (300) C_COMP, BUND IS_A (300) C_BUND.

CONDITION: PART_OF(COMP,BUND,'0020').

RESTRICTION: COMP.XHelper = BUND.Y.

and a procedure $self.X ?= $self.XHelper.

If you have allocated constraints that are meant to relate different items in a bundle to one of the PPO items instead of to the bundle itself we recommend moving the constraint to the bundle level.

Selection conditions in the bundle model do not need adjusting. They are part of the bundle model, not the PPO model.

Advanced variant configuration, AVC, sales order (SET) scenario, bundling, object dependencies
