<objective>
Wire the Analyze toolbar button to the backend `analyzeRisks` action with proper response handling, error display, busy indicator -- using a custom action in manifest.json (NOT DataFieldForAction) per the proven pattern from the prototype.

Purpose: The annotations from Plan 01 provide the visual foundation; this plan adds the interactive behavior. The Analyze button must call the unbound `analyzeRisks` action, process the response to update virtual fields on loaded rows, show a busy indicator during processing, and display errors via MessageBox. This is the critical user-facing interaction that makes the risk analyzer functional.

Output: Fully interactive Fiori Elements list report where clicking Analyze triggers inference, results appear with color-coded criticality, and errors are shown in standard Fiori dialogs.

Note on KPI header (locked decision): The KPI strip with total transaction count and high risk count is a locked decision from CONTEXT.md. However, RESEARCH.md flags dynamic KPI updates (high risk count refreshing after Analyze) as LOW confidence -- it requires either @Aggregation.ApplySupported or a custom header section with controller logic. The static total count could use @UI.KPI but the dynamic high-risk-count update involves SideEffects wiring that Phase 4 addresses. To avoid shipping a half-working KPI that shows static counts but never updates, KPI implementation is deferred to Phase 4 where it can be wired end-to-end alongside SideEffects. This is a sequencing decision, not a scope cut -- the KPI strip WILL be delivered.
</objective>

**Step 4: Create the controller extension** at `app/risks/webapp/ext/controller/ListReportExt.controller.js`:

```javascript
sap.ui.define([
  "sap/m/MessageBox"
], function (MessageBox) {
  "use strict";

  return {
    onAnalyze: function (oBindingContext, aSelectedContexts) {
      // 'this' is the Fiori Elements ExtensionAPI
      var oExtensionAPI = this;
      var oModel = oExtensionAPI.getModel();

      // Get the table reference to set busy state
      // In Fiori Elements V4, use the extension API to get the internal table
      var oTable = oExtensionAPI._controller
        && oExtensionAPI._controller.getView()
        && oExtensionAPI._controller.getView().byId
        ? oExtensionAPI._controller.getView().byId("risk.analysis::GLTransactionsList--fe::table::GLTransactions::LineItem-innerTable")
        : null;

      if (oTable) {
        oTable.setBusy(true);
      }

      // Call unbound action analyzeRisks
      var oActionBinding = oModel.bindContext("/analyzeRisks(...)");

      oActionBinding.execute().then(function () {
        // Action succeeded -- refresh the table binding to show updated virtual fields
        // The analyzeRisks handler returns data but virtual fields need the list binding to refresh
        var oListBinding = oExtensionAPI._controller
          && oExtensionAPI._controller.getView()
          && oExtensionAPI._controller.getView().byId("risk.analysis::GLTransactionsList--fe::table::GLTransactions::LineItem")
          ? oExtensionAPI._controller.getView().byId("risk.analysis::GLTransactionsList--fe::table::GLTransactions::LineItem").getBinding("items")
          : null;

        if (oListBinding) {
          oListBinding.refresh();
        } else {
          // Fallback: refresh the whole model
          oModel.refresh();
        }

        // No success toast per locked decision -- color-coded rows ARE the feedback
      }).catch(function (oError) {
        // Error: show standard Fiori MessageBox.error per locked decision
        // Use i18n-externalized fallback message (UI-05 compliance)
        var sMessage;
        try {
          var oErrorData = JSON.parse(oError.message);
          sMessage = oErrorData.message || oError.message;
        } catch (e) {
          // Fallback to i18n error message -- no hardcoded English strings
          var oResourceBundle = oExtensionAPI.getModel("i18n").getResourceBundle();
          sMessage = oResourceBundle.getText("analyzeError");
        }
        MessageBox.error(sMessage);
      }).finally(function () {
        // Remove busy indicator
        if (oTable) {
          oTable.setBusy(false);
        }
      });
    }
  };
});
```

**IMPORTANT NOTES for the executor:**
- The exact table control ID pattern in Fiori Elements V4 may differ. Query the UI5 MCP server for the correct ID pattern, or use `oExtensionAPI._controller.getView().getContent()` to traverse the view tree. The ID pattern `{appId}::{viewName}--fe::table::{entitySet}::LineItem` is the standard convention.
- If the table ID lookup does not work, use the model refresh fallback (`oModel.refresh()`). This is less optimal (refreshes everything) but reliable.
- The controller is a plain object (NOT an ES6 class extending Controller). This is the Fiori Elements V4 custom action handler pattern -- it receives `this` as the ExtensionAPI.
- Do NOT use `sap.ui.getCore()`, `jQuery.sap.*`, or synchronous loading (AGENTS.md rules).
- Do NOT add a success toast/MessageToast after Analyze (locked decision: no explicit success feedback).
- The error fallback MUST use `oExtensionAPI.getModel("i18n").getResourceBundle().getText("analyzeError")` instead of a hardcoded English string (UI-05 compliance: all user-facing text externalized in i18n).
