<task type="auto">

**MANDATORY: Query the CAP MCP server BEFORE writing any CDS code.** Ask the CAP MCP server about:
- CDS entity syntax for composite keys
- Decimal type precision notation
- Virtual/transient field syntax
- Namespace declaration

Then create `db/schema.cds` with these exact specifications:

```cds
namespace risk;

entity GLTransactions {
  key CompanyCode    : String(4);
  key FiscalYear     : String(4);
  key DocumentNumber : String(10);
  key LineItem       : String(6);

  // GL business fields
  GLAccount   : String(10);
  CostCenter  : String(10);
  PostingDate : Date;
  Amount      : Decimal(15,2);

  // 24 feature columns (snake_case to match BDC/Python conventions)
  anomaly_score      : Decimal(5,4);
  amount_z_score     : Decimal(10,4);
  rarity_score       : Decimal(5,4);
  temporal_score     : Decimal(5,4);
  peer_amount_stddev : Decimal(15,2);
  peer_count         : Integer;
  peer_avg_amount    : Decimal(15,2);
  peer_count_month   : Integer;
  frequency_12m      : Integer;
  is_weekend         : Integer;
  is_after_hours     : Integer;
  is_new_combination : Integer;
  amount_feat        : Decimal(15,2);   // renamed from 'amount' to avoid collision with GL Amount field
  abs_amount         : Decimal(15,2);
  amount_log         : Decimal(10,4);
  peer_amount_ratio  : Decimal(10,4);
  is_large_amount    : Integer;
  posting_delay_days : Integer;
  day_of_week        : Integer;
  posting_hour       : Integer;
  month_numeric      : Integer;
  PostingDate_days   : Integer;
  weekend_and_large  : Integer;
  is_high_frequency  : Integer;

  // Virtual risk result fields (transient, not persisted)
  virtual riskClassification : String;
  virtual riskExplanation    : String;
  virtual anomalyScoreResult : Decimal(5,4);
}
```

**IMPORTANT field naming note:** The feature column named `amount` in FEATURE_COLUMNS collides with the GL business field `Amount`. In the CDS entity, name it `amount_feat` to avoid ambiguity. The FEATURE_COLUMNS constant (Plan 02) maps 'amount' -> 'amount_feat' when extracting from the CDS entity. Similarly, `anomalyScore` virtual field is named `anomalyScoreResult` to avoid collision with `anomaly_score` feature column. Consult the CAP MCP server about whether CDS field names are case-sensitive (they are in SQLite).

Then create `db/data/risk-GLTransactions.csv` with ~50 rows:
- CSV headers MUST exactly match CDS field names (case-sensitive)
- Distribution: ~35 rows Normal (~70%), ~10 rows medium risk (~20%), ~5 rows high risk (~10%)
- SAP-realistic values:
  - CompanyCode: 1000, 2000, 3000
  - FiscalYear: 2024
  - DocumentNumber: 5000000001 through 5000000050
  - LineItem: 000001, 000002
  - GLAccount: 400000, 410000, 500000, 600000, 610000, 700000
  - CostCenter: CC1000, CC2000, CC3000, CC4000
  - PostingDate: dates in 2024 (mostly weekdays)
  - Amount: ranges from 500.00 to 150000.00
- Normal rows (70%): anomaly_score < 0.30, is_weekend=0, is_new_combination=0, small posting_delay_days
- Medium risk rows (20%): anomaly_score 0.30-0.60, is_weekend=1 OR posting_delay_days > 5
- High risk rows (10%): anomaly_score > 0.60, amount_z_score > 3.0, is_new_combination=1
- All amounts positive, clean data (no edge cases)

After creating both files, run `cds compile db/schema.cds` to verify CDS compiles. Then start `cds watch` briefly to confirm CSV data loads and OData endpoint serves data.
